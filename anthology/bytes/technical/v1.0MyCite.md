# Mycite Framework

## Overview
The Mycite framework is a control system and data handling skeleton meant to support the extension to, or development on, of any data and/or software.
The control system design, called 'Control_Gate', is built to interface the management of the in-memory relational data model.

First and foremost, that prioritizes reflecting the program's state as the direct result of instructive input (from a user or otherwise); by reflecting a structure that that of a person’s decision making. External inputs and outputs are automatically processed, but they don’t come into the program's main attention state unless they have some significant value or flag that called for attention. Additionally it aims to allow for a means of navigating in-memory data in a way that reflects sequential steps of a thought process. By allowing for the application user to navigate, manipulate, and mediate on objects of data through the lens of context (context of program state attention, directed intention, chronological position, and the abstractions of in-memory data archetypes that define abstraction to recognize semantic value)

The in-memory relational data model, called the 'Mycelium Schema Standardization', is an asserted supportive platform of a semantic web. Although the schema design is a convention of use, that narrows its capacity, through collaboration or development it is capable of being a convention that can semantically represent any manful ontology of information. Capable of doing so in relation/context to any other manful ontology of information. All while maintaining a system of self organization, under a a single information schema. It sole purpose is hyper-extensibility, however it can be narrowed for uses that better optimize for the demand of different data intensive applications.

This framework allows for applications to be built that allow for users to manage interaction with data as a dynamic and connected mesh that is informed on how it adapts to a users directive, including directives to consider in succession across the interactive system. A model meant to be a single step away from how a robot might contextually operate like a human being, without specific codes, except in this implementation we put the user as the decision input, rather than a closed loop decision.

## Core Components

### main.py

#### AppState
The central brain managing:
- What’s currently focused (attention)
- What’s visible or active (view)
- What relationships or context are linked (earshot, archetypes)
- What intentions or actions are queued for processing

It synchronizes all live system state.

#### Peripheral Processes
These are modular input and output modules:
- StateReadIn → Handles human/device input (keyboard, mouse, sensors)
- StateReadOut → Drives human/device outputs (visual display, feedback)
- MyciteReadIn → Captures network or external system input
- MyciteReadOut → Pushes system state or results outward

Each runs asynchronously, feeding or consuming data for the core system.

#### Intention Handler
This interpretive layer:
- Classifies raw events into four intention types:
    - Navigation → Moving focus or scope
    - Investigation → Querying internal structure
    - Manipulation → Changing data or state
    - Mediation → Coordinating across boundaries
- Dispatches intentions to modify AppState
- Keeps the system’s meaning clear despite multiple asynchronous signals

#### The Main Loop
The synchronous heartbeat:
- Processes queued intentions
- Recalculates the view, state, or outputs
- Drives visual or external updates (e.g., via Pygame or network)

### boot.py

#### The Boot Loader
Taking a layered and modular approach to designing the system's semantic abstraction logic, grounded in the broader principles of the Mycelium Schema Standardization (MSS). 
The core idea is that data should not be interpreted as flat records, but rather as logged addresses—locations within a layered semantic address space. 
The approach begins by unpacking a raw bitstream, which reflects an address-orientation at the lowest level. 
From there, your design separates the reconstruction of schema sections (Index_A, Index_B, Index_C, Index_S, and Index_T) into independently defined parsing functions. 
This mirrors your overarching data model, where abstraction layers (such as SSID Layering) isolate meaning within specific scopes while preserving the ability to reference earlier context. 
Structuring the code this way allows you to evolve each part of the schema interpretation logic independently—honoring the deferred, convention-based semantics of MSS—while maintaining clarity and adaptability as your implementation and control conventions grow.

## How It Works

1. `bootLoad()` Loads system memory (e.g., from .bin files) and unpacks binary data.
2. `bootStrap()` Initializes parsed and structured in-memory relational data base to attach to app state system
3. `main.py` Creates AppState, launches the synchronous main loop.
4. Peripheral Processes: Asynchronously detect inputs or required outputs, adding updates to system buffers.
5. Intention Handler: As a part of the Control Gate convention of implementation; reads primary and secondary peripheral signals, classifies them, and serves them to AppState.
6. StateReadOut: Observes and pushes relevant Appstate updates to inform visuals or outputs to the user or external systems.

## Extensibility Plan
The system is structured to support:
- Adding new input devices or sensor streams
- Supporting HTTP/WebSocket/network protocols (e.g., DNS, BitTorrent)
- Building custom archetype modules for specialized behavior
- Integrating advanced visual/audio interfaces with Pygame or similar libraries
- Supporting multimodal user interaction (navigation, manipulation, mediation)
