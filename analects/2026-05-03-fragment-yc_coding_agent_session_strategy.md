# YC coding-agent session strategy

## Position

The coding-agent attachment should show the technical behavior that FND needs to scale:

- messy input data;
- explicit schema or structural target;
- code generation/refactor/review by agent;
- founder review and correction;
- resulting normalized artifacts;
- validation or audit output.

## Best session type

The strongest session is not a generic frontend build. It is a data-normalization or structured-interface session.

Ideal example:

> Use an AI coding agent to normalize messy farm, voter, GIS, or operational records into structured supply or administrative data, then validate the output with tests, logs, or audit reports.

## Why this fits FND

FND's core technical pattern is converting fragmented domain data into structured operational visibility. In agriculture, that means farm profiles, availability, crop plans, harvest windows, product attributes, and buyer constraints. In current non-agriculture consulting, the same pattern appears as campaign or GIS/voter data normalization.

## YC answer fragment

The best coding-agent session to attach should show a real technical conversion, not a generic UI build: messy domain input, schema normalization, database/API or artifact output, and validation. That maps directly to FND's product: turning fragmented domain records into structured, interoperable data that can support procurement and coordination workflows.
